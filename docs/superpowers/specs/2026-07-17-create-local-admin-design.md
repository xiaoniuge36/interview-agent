# 后台新增管理员设计

## 目标

在现有治理控制台的“账号管理”页面中，允许平台管理员直接创建可登录的本地后台账号，不新增独立系统或页面入口。

## 范围与角色

- 仅当前 platform_admin 可见入口并调用接口。
- 创建角色限定为：
  - platform_admin：固定归入系统租户 system。
  - admin：租户管理员，必须从已有租户中选择归属。
- 不创建普通用户、审核员或客服账号；这些角色仍沿用既有角色调整能力。
- 所有创建账号均为本地账号，使用管理员填写的初始密码。

## 后端设计

新增两个平台管理员专属接口：

- GET /api/admin/accounts/tenants：返回可选择的已有租户，字段为 id、name、slug。
- POST /api/admin/accounts/local-admin：接收姓名、邮箱、初始密码、角色以及租户标识。

创建操作使用既有可串行化事务：

1. 校验平台权限与输入约束。
2. 解析目标租户；平台管理员固定使用 system，租户管理员必须使用存在的 tenantSlug。
3. 对初始密码进行哈希。
4. 创建 User 与唯一的 LocalCredential。
5. 写入 account:local_admin_created 审计日志，仅记录角色、租户和本地认证来源。
6. 返回脱敏后的 AccountView。

邮箱唯一冲突返回业务冲突错误；密码永不出现在响应、审计或列表中。

## 前端设计

在账号管理标题区加入“新增管理员”主按钮，打开 Ant Design Modal：

- 姓名、邮箱、初始密码为必填项。
- 管理员角色为“平台管理员”或“租户管理员”。
- 选择租户管理员时显示必填的已有租户下拉框；选择平台管理员时隐藏该项并由后端固定系统租户。
- 成功后关闭弹窗、重置表单、提示成功并刷新当前账号列表。
- 失败沿用现有全局 API 错误提示；弹窗保留输入，方便修正。

## 验证

- 合约测试验证输入和输出结构。
- 服务测试验证本地凭据、角色、租户、审计与重复邮箱处理。
- 控制台请求测试验证 POST 请求体。
- 组件测试验证现有账号管理页包含入口和管理员表单。
- 运行 Product API 与 Admin Console 的 typecheck、lint、test，并用平台管理员会话进行一次真实创建验收。
