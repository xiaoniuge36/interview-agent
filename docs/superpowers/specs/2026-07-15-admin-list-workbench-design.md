# 后台列表化与审核工作台改造设计

## 背景与问题

当前后台通过单页 hash 切换视图。候选题的 `Table` 位于“题库审核”，而“审核工作台”直接渲染默认候选题的长表单。这让管理者从侧栏进入审核时无法先浏览、筛选或定位待办，并且“题库审核”和“审核工作台”对候选题职责重复。

用户要求后台参考 Ant Design 管理后台范式。此次采用现有自定义组件与 CSS 实现，不新增 `antd` 依赖，也不修改后端 API、contracts、数据库或全局路由。

## 目标

1. “审核工作台”默认展示候选题审核列表：状态筛选、关键词搜索、分页和操作列。
2. 点击“审核”后以右侧 Drawer 打开候选题详情与编辑表单；不再默认选中第一条记录。
3. “题库管理”只展示正式题库，移除重复的候选题队列。
4. “资料导入”以任务列表为主内容，导入表单改为右侧 Drawer 中的创建动作。
5. 统一 Table、Tag、Pagination、Drawer、按钮、间距和响应式行为，使已存在的模型、运行、审计列表与上述页面一致。

## 非目标

- 不新增 `antd` 包或其他依赖。
- 不改 Product API、contracts、数据库、筛选接口或分页接口。
- 不把现有 hash 视图重构为独立 App Router 页面；这是后续可演进项，不阻塞列表化体验。
- 不为模型、运行观测、审计日志凭空添加编辑操作。

## 信息架构

| 导航项     | 页面职责         | 默认形态  | 主要操作              |
| ---------- | ---------------- | --------- | --------------------- |
| 治理总览   | 指标与待办入口   | Dashboard | 跳转到对应模块        |
| 资料导入   | 导入任务管理     | Table     | “导入资料”打开 Drawer |
| 题库管理   | 已发布/正式题库  | Table     | 筛选与检索            |
| 审核工作台 | 候选题治理       | Table     | “审核”打开 Drawer     |
| 模型治理   | 模型配置只读治理 | Table     | 筛选与检索            |
| 运行观测   | Agent 运行记录   | Table     | 筛选与检索            |
| 审计日志   | 操作追溯         | Table     | 筛选与检索            |

## 交互设计

### 审核工作台

进入页面时仅显示候选题列表。列表的操作列为“审核”；选择一行后打开右侧 Drawer，并用既有 `GET /admin/candidates/:id` 读取详情。抽屉内保留现有保存审核和发布操作。关闭抽屉只清除选择，不做隐式保存。

候选题选择必须是显式行为：无论从侧栏、总览还是刷新后的列表进入，都不能回退到数组中的第一条候选题。若已选记录在刷新后消失，Drawer 自动关闭。

### 资料导入

页面顶部显示标题、说明和“导入资料”主按钮，下方为导入任务表格。点击按钮后使用同一套 Drawer 承载现有 Markdown 表单；提交成功后刷新任务列表并保留明确的状态反馈。

### 组件语言

保留语义化 HTML 表格和原有图标，使用 Ant Design 的结构与尺度作为参考：白底 8px Card、`#1677ff` 主操作、`#fafafa` 表头、低强调边框、紧凑 32px/40px 控件、状态 Tag 和连续页码 Pagination。Drawer 在桌面端从右侧滑出，在窄屏占满可用宽度。

## 状态与数据流

```text
候选题列表 -> 点击“审核” -> selectedCandidateId
                                  |
                                  v
                           AdminDrawer 打开
                                  |
                                  v
                         CandidateEditor 详情请求
                                  |
                         保存 / 发布成功 -> reload()
                                  |
                                  v
                         列表重新加载，保持或关闭显式选择
```

`AdminDrawer` 只处理展示、关闭和 Escape 键；候选题与导入各自保留已有的请求、错误和成功提示逻辑。所有筛选及分页仍在前端内存中完成，符合当前 API 响应结构。

## 验收标准

- 进入“审核工作台”时，不显示候选题编辑表单，先显示传统列表。
- 点击候选题“审核”后才出现带标题的右侧 Drawer 和编辑表单。
- 不存在默认选择第一条候选题的回退逻辑。
- “题库管理”不再显示候选题审核队列。
- “资料导入”首屏为导入任务列表，创建表单在 Drawer 内。
- 模型、运行与审计列表保持可用，并获得统一的分页和表格视觉。
- admin-console 的单元测试、lint、typecheck、build 通过；在本地可启动环境中完成关键路径浏览器验证。

## 风险与缓解

- 现有工作区已有未提交的后台壳层与 token 改动：本次只在其基础上增量修改，不重置、不覆盖无关变更。
- 前端当前为 hash 视图且一次性加载数据：本次不重构路由或 API，以避免超出 UI 修复范围；后续可单列服务端分页与真实路由工作。
- Drawer 不会引入第三方焦点管理库：实现 Escape 关闭、可访问的 `dialog` 语义、显式关闭按钮和遮罩点击关闭，并在浏览器验证键盘路径。
