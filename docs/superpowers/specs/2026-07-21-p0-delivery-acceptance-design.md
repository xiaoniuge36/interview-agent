# P0 交付闭环验收设计

## 目标

为既有的用户端与后台管理端建立可重复、无真实模型费用的浏览器验收门禁，覆盖最关键的业务闭环与已知失败路径；同时修正文档对模型能力的过期描述。此次不新增业务系统、数据库表、接口契约或生产模型凭证。

## 已确认边界

- 用户端、后台、Product API 与 Agent Runtime 已能本地协同启动；现有 CI 已执行迁移、单元/集成测试、构建、安全扫描和 Docker 构建，但没有浏览器 E2E。
- 模型调用必须仍经过 Product API 的凭证解析、加密存储、用量记录与错误映射，浏览器或测试脚本不得持有用户 API Key。
- E2E 不调用公共模型提供商。测试环境启动一个仅监听本机回环地址的 OpenAI-compatible HTTP 替身，返回固定、可验证的 JSON/SSE 响应。
- E2E 不覆盖所有细节组件测试；它只验证高价值的跨服务闭环与阻断性失败路径。已有 Vitest/Jest/Pytest 继续负责细粒度行为。

## 方案选择

### 方案 A：仅保留单元和 API 集成测试

不增加浏览器测试，成本最低，但无法发现登录态、路由、表单、SSE 和跨应用配置的真实断链。拒绝。

### 方案 B：独立 Playwright 验收包（采用）

在根工作区增加 Playwright，并使用测试专用启动脚本拉起 PostgreSQL/Redis、Product API、Agent Runtime、两个 Next 应用和本机模型替身。测试通过本地认证页面和真实 HTTP 接口完成操作。优点是测试边界接近用户体验、可在 CI 复现；代价是 CI 时长和启动编排增加。

### 方案 C：对外真实模型冒烟

验证最接近生产，但依赖外网、密钥、费用与模型响应稳定性，不适合作为提交门禁。仅可作为受控环境的人工发布前检查，不能替代方案 B。

## 测试架构

```text
Playwright
  -> User Portal :3000 / Admin Console :3002
       -> Product API :3001
            -> Agent Runtime :8000
                 -> Product API internal model gateway
                      -> local model stub
```

测试使用单独数据库 schema 和独立本地端口。每个 suite 启动前通过现有迁移、种子和 HTTP 清理接口（若已有）恢复确定状态；若没有安全的清理接口，则由测试脚本在隔离 schema 中执行 Prisma 数据删除。绝不清理开发者正在使用的默认数据库。

模型替身必须支持：

1. 兼容模型完成请求，返回固定的合格结构化评价与面试决策；
2. 流式评价请求，按固定顺序发送可见 feedback 增量与最终 usage；
3. 受测试开关控制的超时、429 与非法 JSON，以验证传统错误提示、重试和不产生半成品报告。

## 必测旅程

### 用户端

1. 本地注册/登录后能进入受保护应用；清除令牌或收到 401 后回到登录页。
2. 创建或选择目标岗位/JD，打开题目页并开始推荐练习。
3. 保存全部答案，确认“将消耗模型额度”后生成整轮复盘；检查报告、掌握度与下一轮推荐均可见。
4. 单题评价的 SSE 显示增量反馈，最终结果替换临时状态且不会重复。
5. 模型不可用或结构化结果非法时，显示面向用户的传统错误提示，已保存答案不丢失，报告不进入完成态。

### 后台

1. 管理员登录后看到治理控制台；普通用户无法越权访问。
2. 创建模型连接并测试连接，密钥在后续读取中不可见。
3. 创建导入任务，定位候选题，审核通过后发布到题库；候选题状态、已发布题和审计/看板指标一致。
4. 运行观测和 AI 用量页能展示模型、用户、token、时延和失败原因的脱敏数据。

## CI 与运行策略

- 新增 `test:e2e` 根脚本，默认本地显式运行；CI 增加一个独立 job，等待依赖服务健康后运行 Chromium E2E。
- E2E job 失败时保留 Playwright trace、截图和视频作为构件；敏感环境变量与模型请求正文不得写入构件。
- 将 E2E 的总超时限制为 12 分钟；每条测试使用稳定的 `data-testid` 或可访问名称，不依赖布局像素、随机文案或真实时间。
- 现有质量 job 保持不变；E2E 只在基础质量门禁成功后执行。

## 文档与发布治理

- README 如实说明：面试主链已经接入受控的真实 Provider 调用；RAG、向量检索、评测集、LLM judge、集中式观测与生产云部署仍未交付。
- README 新增“发布前验证”章节，列出本地 E2E、生产认证/OIDC、密钥轮换、模型连接测试、健康检查和回滚负责人的检查项。
- 不在仓库中写入任何生产密钥、成本单价或真实用户数据。

## 非目标

- 不在本 P0 新增 RAG、MCP、长时任务队列、额度计费、反馈数据表或新的后台应用。
- 不将真实生产模型作为 CI 依赖。
- 不重置、覆盖或混入当前工作区已有的用户端刷题教练改动。

## 验收标准

1. 从干净依赖环境运行 `pnpm test:e2e` 可以完整启动隔离服务并通过用户端和后台关键路径。
2. CI 能运行同一命令并在失败时上传可排查构件。
3. 成功、401、模型不可用和非法模型 JSON 四类结果可稳定复现。
4. README 与实际代码能力一致，不再宣称真实 LLM 主链“尚未接入”。
